## Определение системы

Система — это группа кода, которая:
- реализует конкретную игровую или бизнес-механику
- имеет вход (input), обработку (logic), выход (output)
- используется более чем в одном месте ИЛИ должна использоваться повторно

### Примеры:
- система урона
- система бросков кубов
- система эффектов/баффов
- система ресурсов (мана, стамина)
- система инвентаря

## Задача анализа проекта

1. Найти все системы в проекте

2. Для каждой системы определить:
   - Название
   - Назначение
   - Где находится (файлы)
   - Основные функции/методы
   - Какие данные использует
   - С чем взаимодействует

3. Найти:
   - Дублирующуюся логику
   - Повторяющиеся паттерны
   - "Размазанные" системы (логика разбросана по разным файлам)
   - Места, где система должна быть, но её нет


## Как выявлять неявные системы

Считай, что система существует, если:

- Один и тот же код повторяется в 2+ местах
- Одинаковые проверки/условия используются в разных файлах
- Одинаковые структуры данных создаются вручную
- Есть логика, связанная общей темой (например: урон, лечение, эффекты)
- Есть цепочки вызовов, которые всегда идут вместе

### Примеры сигналов:
- "rollDice()" вызывается в 5 местах → система бросков
- "if (hp <= 0)" в разных местах → система смерти/состояния
- одинаковые списки эффектов → отсутствует централизованная система эффектов

## Формат отчёта

### Система: <название>

**Описание:**
Кратко, что делает система

**Файлы:**
- path/to/file1
- path/to/file2

**Ключевые функции:**
- functionA()
- functionB()

**Используемые данные:**
- hp
- damage
- effects

**Зависимости:**
- зависит от: ...
- используется в: ...

**Проблемы:**
- дублирование: (где именно)
- разбросанность: (где)
- отсутствие абстракции

---

### Потенциальная система (не выделена)

**Описание:**
Что выглядит как система, но не оформлено

**Где найдено:**
- file1
- file2

**Почему это система:**
- повторяется X раз
- выполняет одинаковую функцию

**Рекомендация:**
- вынести в отдельный модуль

## Ограничения

- Не выдумывать системы, которых нет в коде
- Основываться только на реальном использовании
- Если не уверен — помечать как "гипотеза"
- Не описывать UI как систему, если там нет логики

## Классификация систем

Для каждой системы укажи тип:

- Core (ядро механики)
- Utility (вспомогательная)
- Data (работа с данными)
- UI (интерфейс)
- Glue (связывает другие системы)

## Оценка качества

Оцени каждую систему:

- Связанность (low / medium / high)
- Дублирование (yes / no)
- Централизация (low / medium / high)

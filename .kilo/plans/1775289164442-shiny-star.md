# Архитектурный отчёт — Vitruvium

## 1. Системы (механики)

### 1.1. Ядро системы (System Bootstrap)

**Назначение:** Инициализация системы, регистрация листов, настроек, хуков, Handlebars-хелперов.

**Файлы:**
- `module/system.js`

**Ключевые функции:**
- `Hooks.once("init", ...)` — точка входа
- Регистрация всех ActorSheet и ItemSheet
- `game.settings.register("enableAutomatedAnimations")`
- `CONFIG.Dice.terms["V"] = VitruviumDie`
- Хелперы Handlebars: `inc`, `grouped_options`
- Миграция NPC-атрибутов (flat → `{value: X}`)
- Хуки `preCreateActor`, `preUpdateActor`, `updateActor`, `preCreateToken` — синхронизация токенов с актёрами
- `game.vitruvium` API: `startAbilityAttackFlow`, `startWeaponAttackFlow`, `replaceStateFromTemplate`, `showFloatingText`

**Зависимости:** Все модули (импортирует все листы, combat, initiative, tests, state-duration, floating-text, chat-drop, dv-die, dice-so-nice)

### 1.4. Система бросков (Roll System)

**Назначение:** Броски пула dV с подсчётом успехов (1-3=0, 4-5=1, 6=2), преимущество/помеха, полный переброс (удачливый/неудачливый).

**Файлы:**
- `module/rolls.js`

**Ключевые функции:**
- `dvSuccesses(face)` — подсчёт успехов одного куба
- `dvFaceKind(face)` — тип грани: blank/single/double
- `renderFaces(results)` — HTML-рендер граней
- `rollDieOnce(roller)` — бросок 1dV
- `pickIndex(results, preferHighest)` — выбор худшего/лучшего куба для переброса
- `rollSuccessDice({...})` — основной API броска с pool/luck/unluck/fullMode/roller

**Зависимости:** `chat-visibility.js`

---

### 1.5. Система эффектов (Effects System)

**Назначение:** Агрегация модификаторов из предметов/способностей/состояний. Нормализация legacy-ключей, сборка totals, вычисление модификаторов бросков.

**Файлы:**
- `module/effects.js`

**Ключевые функции:**
- `EFFECT_TARGETS` — реестр всех возможных ключей эффектов (атрибуты, броски, защита, общие модификаторы, характеристики)
- `LEGACY_EFFECT_KEY_MAP` — маппинг legacy ключей (rollAdv/rollDis → rollLuck со знаком)
- `normalizeEffects(raw)` — нормализация, агрегация, legacy-конверсия, overtime-эффекты (dot/hot)
- `collectEffectTotals(actor)` — суммирование всех эффектов актёра с учётом equipped/active
- `getEffectValue(totals, key)` — получение значения эффекта
- `getLuckModifiers(totals, {signedKey, advKey, disKey})` — разбор luck в adv/dis
- `getEffectiveAttribute(attrs, key, totals)` — атрибут с эффектами
- `getGlobalRollModifiers(totals)` — глобальные модификаторы всех бросков
- `getAttributeRollModifiers(totals, attrKey)` — модификаторы броска конкретного атрибута
- `getAttackRollModifiers(totals, {attrKey})` — модификаторы броска атаки
- `openEffectsDialog(item)` — диалог редактирования эффектов
- `OVERTIME_EFFECT_TYPES` — dot/hot
- `OVERTIME_TRIGGER_TIMINGS` — start/end

**Зависимости:** Нет внешних

---

### 1.6. Система боя (Combat System)

**Назначение:** Полный цикл боя — атака оружием, атака способностью, защита (уклонение/блок), состязание состояний, расчёт урона, криты, GM-only кнопки применения урона/лечения.

**Файлы:**
- `module/combat.js` (2828 строк — самый большой файл)

**Ключевые функции:**

*Утилиты:*
- `dvSuccesses(face)`, `dvFaceKind(face)`, `clamp()`, `num()`, `esc()` — дублируются из rolls.js
- `pickIndex()`, `rollDieOnce()` — дублируются из rolls.js
- `modeLabel()`, `fullModeLabel()` — дублируются из initiative.js

*Броски:*
- `rollPool(pool, opts)` — аналог `rollSuccessDice`, но со своей логикой (дублирование!)

*Диалоги:*
- `attackDialog({...})` — выбор атрибута атаки, luck/unluck/fullMode
- `defenseDialog({...})` — выбор уклонение/блок
- `contestDialog({...})` — выбор атрибута сопротивления

*Карточки чата:*
- `renderCollapsibleBox({...})`
- `attackCardTwoCols({...})`
- `defenseCardTwoCols({...})`
- `abilityAttackCard({...})`
- `contestRollCardTwoCols({...})`
- `resolveContestCardHTML({...})`
- `resolveCardHTML({...})`, `resolveAbilityCardHTML({...})`
- `renderDefenseTargets({...})`, `renderContestTargets({...})`
- `renderGmApplyButtons({...})`

*Расчёт урона:*
- `isCriticalHit(atk, def)` — крит: atk >= 2 && atk >= 2*def && atk - def >= 2
- `computeDamageCompact({...})` — урон оружия (block/dodge формулы)
- `computeAbilityDamage({...})` — урон способности
- `getWeaponDamage()`, `getWeaponRollMods()`, `getArmorTotal()`, `hasHeavyArmorEquipped()`, `hasBlockWeaponEquipped()`

*Состояния:*
- `replaceStateFromTemplate(defenderActor, templateUuid, durationOverride, defenderTokenUuid)` — создание состояния из шаблона

*Публичный API:*
- `startAbilityAttackFlow(attackerActor, abilityItem)` — полный цикл атаки способностью
- `startWeaponAttackFlow(attackerActor, weaponItem)` — полный цикл атаки оружием

*Хуки:*
- `renderChatMessageHTML` — биндинг кнопок защиты, сопротивления, применения урона
- `createChatMessage` (GM) — обработка contest resolve requests

*Флаги:*
- `FLAG_SCOPE_PRIMARY = "Vitruvium"`, `FLAG_SCOPE_LEGACY = "vitruvium"` — двойная запись флагов
- `SOCKET_EVENT_GM_APPLY`, `SOCKET_NAMESPACE_PRIMARY/LEGACY` — сокеты для GM

**Зависимости:** `auto-animations.js`, `effects.js`, `state-library.js`, `chat-visibility.js`

## 3. Неявные системы (разрозненные куски)

### 3.1. Система атрибутов
Атрибуты (condition, attention, movement, combat, thinking, communication) упоминаются и обрабатываются в:
- `template.json` — определение
- `system.js` — миграция
- `effects.js` — `ROLL_ATTRIBUTE_KEYS`, `EFFECT_TARGETS`
- `combat.js` — `listAttributeKeys`, `prettyAttrLabel`, `getEffectiveAttribute`
- `character-sheet.js` — рендер, инкремент/декремент, броски
- `npc-sheet.js` — рендер, инкремент/декремент, броски
- `initiative.js` — movement для инициативы
- `item-sheet.js`, `ability-sheet.js` — attackAttr options

**Нет единого модуля атрибутов.**

### 3.2. Система HP
HP логика разбросана по:
- `template.json` — `{value, max}`
- `system.js` — миграция
- `character-sheet.js` — вычисление hpMax = condition * 8 + hpMax effect, debounce input
- `npc-sheet.js` — чтение напрямую
- `combat.js` — применение урона/лечения
- `state-duration.js` — DoT/HoT
- `floating-text.js` — отслеживание изменений

### 3.3. Система монет
Только в `character-sheet.js` — bronze/silver/gold с автосохранением. Нет в NPC. Нет отдельного модуля.

### 3.4. Система вдохновения (Inspiration)
Логика в `character-sheet.js` и `npc-sheet.js` — инкремент/декремент с эффектами. Дублируется.

### 3.5. Token-Actor синхронизация
Хуки `preCreateActor`, `preUpdateActor`, `updateActor`, `preCreateToken` в `system.js` — неявная система синхронизации.

### 3.6. Socket-система
Сокеты используются в:
- `combat.js` — `SOCKET_EVENT_GM_APPLY`, `SOCKET_NAMESPACE_PRIMARY/LEGACY`
- `initiative.js` — `vitruvium-roll-initiative`
- `floating-text.js` — `floatingText`

Нет единого socket-менеджера. Каждый модуль регистрирует свои обработчики.

### 3.7. NPC Migration
Миграция NPC-атрибутов в `system.js:23-82` — одноразовый хук на `ready`. Не отделена в миграционный модуль.

## 4. Отсутствующие абстракции

### 4.1. Нет общего dice/roll utils модуля
`dvSuccesses`, `dvFaceKind`, `rollDieOnce`, `pickIndex`, `clamp`, `num`, `esc`, `modeLabel`, `fullModeLabel` — всё это должно быть в одном `utils/dice.js` или `utils/common.js`.

### 4.2. Нет абстракции для rollPool
Две разные реализации `rollPool` (combat.js и initiative.js) должны быть одной функцией с опциональным dice3d callback.

### 4.3. Нет абстракции для dialog-паттерна
Все диалоги (attackDialog, defenseDialog, contestDialog, rollModeDialog) имеют одинаковую HTML-структуру и логику. Нужен `createRollDialog(opts)`.

### 4.4. Нет абстракции для chat card rendering
Все функции `*Card*` генерируют HTML вручную. Нужен шаблонизатор или builder.

### 4.5. Нет абстракции для effects table UI
Effects table рендерится и обрабатывается идентично в 4 файлах. Нужен общий компонент.

### 4.6. Нет абстракции для sheet base class
Все листы дублируют: tab switching, description edit toggle, `_saveDescOnClose`, immediate-save паттерны. Нужен `VitruviumBaseSheet` или миксины.

### 4.7. Нет единого модуля атрибутов
Атрибуты — центральная концепция системы, но нет модуля, который бы инкапсулировал: keys, labels, icons, roll logic, effective value calculation.

### 4.8. Нет абстракции для socket-коммуникации
Каждый модуль сам регистрирует socket handlers. Нужен `socket-manager.js`.

### 4.9. Нет абстракции для combat flags
Чтение/запись флагов (`FLAG_SCOPE_PRIMARY`, `FLAG_SCOPE_LEGACY`) дублируется. Нужен `CombatFlags` класс.

### 4.10. Нет абстракции для contest states normalization
Логика нормализации contestStates (migration, validation) дублируется в 3 местах.

## 8. Критические замечания

1. **combat.js — 2828 строк.** Это God Object. Содержит: утилиты, диалоги, карточки, расчёт урона, сокеты, хуки, публичный API, нормализацию состояний. Должен быть разбит на 5-7 модулей.

2. **Две реализации rollPool.** `combat.js:rollPool` и `initiative.js:rollPool` — разные функции с одинаковой логикой. Это источник багов.

3. **Двойная запись флагов.** `FLAG_SCOPE_PRIMARY` и `FLAG_SCOPE_LEGACY` — запись в оба скопа при каждом обновлении. Это костыль для обратной совместимости, который нужно убрать после миграции.

4. **VitruviumSkillSheet зарегистрирован дважды** — для `skill` и `state`. Это работает, но создаёт путаницу.

5. **Нет разделения на client/GM код.** Весь код загружается на всех клиентах, хотя большая часть combat.js нужна только GM.

6. **Inline CSS в combat.js:1473** — стили чат-карточек встроены в JS через `document.createElement("style")`. Должны быть в CSS файле.
